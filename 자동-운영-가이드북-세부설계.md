# 자동 운영 가이드북 생성 시스템 — 세부설계

> 2026-07-08 · 대상: 사내 개발/운영팀(~10명) · 기간: 3주(7월말) MVP
> 스택: AWS Bedrock(Claude) + Supabase(pgvector) 확정 기준

---

## 1. 개요

### 1.1 목적
신규 프로젝트의 운영 문서(runbook)를 소스코드·설계문서·ERD·Jira/Confluence로부터 **자동 생성**하고, 생성된 지식 기반 위에서 **출처가 표기된 Q&A**를 제공한다.

### 1.2 범위 (MVP)
- 프로젝트별 소스 등록: Git 레포, Confluence 스페이스, Jira 프로젝트, 문서 파일 업로드
- 일 단위 갱신 감지 및 증분 재인덱싱 (Git)
- 가이드북 자동 생성: 목차/키워드 구조 생성 → 사용자 확인 → 섹션별 본문 생성
- 질의 화면: 정해진 응답 형식(도메인 설명·확인 항목·처리 방안·업무 플로우·출처)으로 답변
- 저작 도구: 생성된 섹션 편집/승인, 프로젝트·소스 관리

### 1.3 비범위 (MVP 제외 → 백로그)
- SSO/전사 권한 체계 (사내 소수 사용 전제, Supabase Auth 초대제로 충분)
- Confluence/Jira 실시간 웹훅 동기화 (일 배치로 시작)
- 가이드북 다국어, PDF 내보내기, 코드 실행형 runbook

### 1.4 핵심 검증 포인트 (기획 메모의 "생각해볼점")
**출처 표기 가능한가 → 가능.** 청크 단위로 원본 퍼머링크(Git blob URL+commit, Confluence page URL, Jira issue key)를 메타데이터로 저장하고, LLM에 청크별 인용 ID를 부여해 응답에서 `[1]`, `[2]`로 인용시킨 뒤 UI에서 링크로 매핑한다. (§5.4)

│   · Runbook 생성 잡 트리거 · CRUD                        │
└───────┬───────────────────────────┬────────────────────┘
        │                           │
┌───────▼────────────┐    ┌─────────▼──────────────┐

### 2.1 구성 결정과 근거
| 항목 | 결정 | 근거 |
|---|---|---|
| 웹/API | Next.js 단일 앱 (App Router) | 3주 MVP, 화면+API 한 코드베이스. Supabase SDK 궁합 |
| 인제스천 | 별도 워커 컨테이너 (ECS Fargate 또는 EC2 docker 1대) | Git clone은 파일시스템·장시간 실행 필요 → Edge Function 부적합 |
| 벡터 저장 | Supabase pgvector (HNSW 인덱스) | 별도 벡터DB 불필요 규모 (수만~수십만 청크) |
| 생성 모델 | Bedrock `claude-sonnet` (답변·섹션 생성), `claude-haiku` (목차 키워드 추출·요약) | 품질/비용 분리 |
| 임베딩 | Bedrock **Cohere Embed Multilingual** (1024차원) | 한국어 문서 중심. 대안: Titan Embed v2 |
| 스트리밍 | SSE | 챗 응답 체감 속도 |

### 2.2 배포
- Next.js: Vercel 또는 ECS. 사내용이면 ECS + ALB(사내망 제한) 권장
- Worker: 동일 VPC의 Fargate 스케줄드 태스크(일 1회) + 온디맨드 실행(수동 동기화 버튼)
- 시크릿: AWS Secrets Manager (Git 토큰, Atlassian API 토큰, Supabase 키)

## 3. 데이터 모델 (Supabase / Postgres)

```sql
-- 프로젝트 & 소스
project        (id uuid PK, name, description, created_by, created_at)
source         (id uuid PK, project_id FK, type,          -- git|confluence|jira|file
                config jsonb,                              -- 레포 URL/브랜치, 스페이스 키, JQL 등
                sync_cron text default '0 6 * * *',        -- 일 갱신
                last_synced_at, status)                    -- active|error|paused

-- 수집 문서 & 청크
document       (id uuid PK, source_id FK, external_id,     -- 파일경로|pageId|issueKey
                title, url,                                -- 출처 퍼머링크
                doc_type,                                  -- code|markdown|confluence|jira|erd|file
                content_hash, version_label,               -- commit sha, confluence version 등
                updated_at, deleted boolean)
chunk          (id uuid PK, document_id FK, seq int,
                content text, token_count int,
                heading_path text[],                       -- ["운영", "배포", "롤백"]
                embedding vector(1024),
                metadata jsonb)                            -- line range, issue status 등
-- 인덱스: HNSW(embedding vector_cosine_ops), GIN(content gin_trgm_ops)

-- 가이드북
runbook        (id uuid PK, project_id FK, title, status,  -- draft|generating|review|published
                toc jsonb,                                 -- 목차 트리 + 섹션별 키워드
                version int, created_at)
runbook_section(id uuid PK, runbook_id FK, order_no, title,
                keywords text[], content_md text,
                citations jsonb,                           -- [{n:1, chunk_id, url, title}]
                status,                                    -- generated|edited|approved
                edited_by, updated_at)

-- Q&A
chat_session   (id uuid PK, project_id FK, user_id, created_at)
chat_message   (id uuid PK, session_id FK, role, content,
                structured jsonb,                          -- 응답 형식 §5.3 원본
                citations jsonb, created_at)

-- 운영
sync_job       (id uuid PK, source_id FK, trigger,         -- cron|manual
                status, stats jsonb,                       -- 신규/변경/삭제/임베딩 건수
                error text, started_at, finished_at)
```

원칙: 프로젝트 간 격리는 RLS(`project_id` 기준)로. 삭제된 원본 문서는 `deleted=true` 소프트 삭제 후 검색에서 제외.

## 4. 인제스천 파이프라인

### 4.1 공통 흐름
```
fetch → normalize(→ markdown) → chunk → hash diff → embed(변경분만) → upsert → sync_job 기록
```
- **증분 갱신**: `document.content_hash` 비교로 변경 문서만 재청킹·재임베딩. Git은 `git diff --name-status <last_sha>..HEAD`로 변경 파일만 처리 → 임베딩 비용 최소화
- **건별 예외 격리**: 문서 1건 실패가 배치 전체를 중단시키지 않도록 try/catch + `sync_job.stats.failures`에 기록
- **삭제 반영**: 원본에서 사라진 external_id는 soft delete

### 4.2 소스별 커넥터
| 소스 | 수집 방법 | 정규화 | 출처 URL |
|---|---|---|---|
| Git 레포 | shallow clone + daily pull. 대상: `*.md`, `*.sql`(DDL/ERD), 소스코드(주석·시그니처 중심), README, docs/ | 코드는 파일 단위 → 클래스/함수 시그니처+주석 요약(Haiku) 후 인덱싱. 문서는 원문 | `{repoUrl}/blob/{sha}/{path}#L{n}` |
| Confluence | REST API v2, 스페이스 단위. `version.number` 변경분만 | storage format(XHTML) → markdown 변환 | page `_links.webui` |
| Jira | REST API, JQL(`project=X AND updated >= -1d`). 이슈 요약+설명+해결 코멘트 | 이슈 1건=문서 1건. 상태/해결책 메타데이터 보존 | `{base}/browse/{KEY}` |
| 파일 업로드 | Supabase Storage 업로드 → 즉시 인덱싱. md/pdf/docx/이미지(ERD) | pdf/docx → 텍스트 추출. ERD 이미지 → Claude 비전으로 테이블·관계 서술 텍스트화 | Storage signed URL |

> 메모의 "문서 파일 — 질의에서 기억할 수 있는 기능?" → 업로드 즉시 청킹·임베딩되어 이후 모든 질의에서 검색됨. 별도 "기억" 기능 불필요.

### 4.3 청킹 전략
- 마크다운/위키: **헤딩 경계 우선** 분할, 청크 500~800 토큰, 오버랩 80 토큰, `heading_path` 보존(검색 컨텍스트 강화)
- 코드: 파일 요약 1청크 + 주요 클래스/함수별 1청크
- Jira: 이슈당 1~2청크 (요약+본문 / 해결 코멘트)

## 5. 검색(RAG)과 Q&A

### 5.1 하이브리드 검색
```
후보 = pgvector cosine top-20  ∪  pg_trgm 키워드 top-20  (프로젝트 필터 + deleted 제외)
     → RRF(Reciprocal Rank Fusion)로 병합 → 상위 8~10개를 프롬프트에 투입
```
한국어 형태소 기반 FTS는 Postgres 기본 미지원 → `pg_trgm`으로 대체(에러코드·클래스명 같은 exact 키워드에 강함). 품질 미달 시 Bedrock Rerank 추가(백로그).

### 5.2 Q&A 흐름
1. 질문 수신 → (멀티턴이면 Haiku로 질문 재작성) → 하이브리드 검색
2. 청크에 인용 번호 부여해 프롬프트 구성 → Sonnet에 **응답 형식 스키마(§5.3)** 로 structured output 요청, SSE 스트리밍
3. 응답의 인용 번호를 chunk→document.url로 매핑해 출처 렌더링, `chat_message`에 저장

### 5.3 응답 형식 (기획 메모의 스키마를 JSON으로 고정)
```json
{
  "query_summary":  "질의 요약(재해석)",
  "domain_brief":   "관련 도메인 간략 설명",
  "checklist":      ["확인해야 할 사항/항목", "..."],
  "resolution":     "처리 방안 (단계별)",
  "workflow":       ["업무 플로우 step 1", "step 2"],
  "citations":      [{"n": 1, "title": "...", "url": "...", "source_type": "confluence"}],
  "confidence":     "high|medium|low",
  "not_found":      false
}
```
- 근거 청크가 없으면 `not_found: true` + "문서에 근거 없음"을 명시 (환각 방지)
- 프롬프트 규칙: "제공된 청크에 있는 내용만 답하고, 모든 주장 뒤에 [n] 인용"

### 5.4 출처 표기 구현 (핵심)
- 저장 시점: 청크 메타데이터에 퍼머링크·버전(commit sha, page version) 고정 → 문서가 갱신돼도 답변 당시 근거 추적 가능
- 생성 시점: 청크마다 `[n]` ID를 부여한 컨텍스트를 주고 인용 강제
- 표시 시점: UI에서 `[n]` → 출처 카드(제목·유형 아이콘·링크)로 렌더링

## 6. Runbook 자동 생성 파이프라인

3단계 — **구조 생성 → 사용자 확인 → 섹션별 생성** (메모의 "목차 및 키워드 구조 생성 확인" 반영)

### 6.1 Stage 1: 목차/키워드 구조 생성
1. 프로젝트 전 문서의 `title + heading_path + 요약`을 수집 (본문 아님 → 토큰 절약)
2. Sonnet에 투입 → 목차 트리 생성. 각 섹션에 `keywords[]`(검색용) 부여
3. 기본 골격 템플릿을 시드로 제공: 시스템 개요 / 도메인별 설명 / 배포·인프라 / 모니터링·알림 / 장애 대응 시나리오 / 정기 운영 작업 / FAQ·트러블슈팅
4. `runbook.toc`에 저장, 상태 `draft`

### 6.2 Stage 2: 사용자 확인 (저작도구)
- 목차 트리 UI에서 섹션 추가/삭제/이름변경/순서변경/키워드 수정
- "생성 시작" 버튼 → 상태 `generating`

### 6.3 Stage 3: 섹션별 본문 생성
- 섹션마다: `keywords + 섹션 제목`으로 하이브리드 검색(top 10~15) → Sonnet이 출처 인용 포함 마크다운 생성 → `runbook_section` 저장(상태 `generated`)
- 섹션 단위 비동기 잡(순차 실행, Bedrock rate limit 보호). 실패 섹션만 개별 재생성 가능
- 근거 부족 섹션은 본문 대신 `⚠ 근거 문서 부족` 배너 + 필요한 문서 제안

### 6.4 갱신 정책 (MVP)
- 자동 재생성 없음. 소스 동기화 후 "이 섹션의 근거 문서가 갱신됨" 뱃지만 표시(citations의 chunk 갱신 여부로 판정) → 사용자가 섹션별 재생성 버튼 클릭
- 재생성 시 `version` 증가, 이전 본문은 편집 이력으로 보존

## 7. 화면 설계 (대시보드)

| # | 화면 | 핵심 요소 |
|---|---|---|
| 1 | 프로젝트 목록 | 카드형. 프로젝트 추가, 최근 동기화 상태, runbook 상태 |
| 2 | 프로젝트 설정 — 소스 연결 | 소스 유형별 등록 폼(Git URL/브랜치/토큰, Confluence 스페이스, Jira JQL), 파일 업로드 존, 소스별 동기화 이력·수동 동기화 버튼 |
| 3 | 가이드북 뷰어 | 좌: 목차 트리, 중: 본문(출처 각주 링크), 우: 키워드·근거 문서 패널. "근거 갱신됨" 뱃지 |
| 4 | 가이드북 저작 | 목차 편집(Stage 2), 섹션 마크다운 에디터, 섹션 상태(generated→edited→approved), 섹션 재생성 |
| 5 | 질의 챗 | 프로젝트 선택 → 챗. 응답은 §5.3 구조 렌더링(도메인 설명/체크리스트/처리방안/플로우 카드 + 출처), 스트리밍 |
| 6 | 동기화 현황 (관리) | sync_job 목록, 실패 문서, 인덱싱 통계(문서/청크 수) |

## 8. API 설계 (주요 엔드포인트)

```
POST   /api/projects                          프로젝트 생성
POST   /api/projects/:id/sources              소스 등록 (type+config)
POST   /api/sources/:id/sync                  수동 동기화 트리거
GET    /api/sources/:id/jobs                  동기화 이력

POST   /api/projects/:id/runbooks             생성 시작 (Stage 1)
PUT    /api/runbooks/:id/toc                  목차 확정/수정 (Stage 2)
POST   /api/runbooks/:id/generate             본문 생성 시작 (Stage 3)
POST   /api/sections/:id/regenerate           섹션 재생성
PUT    /api/sections/:id                      섹션 편집/승인

POST   /api/projects/:id/chat                 질의 (SSE 스트리밍, §5.3 구조 응답)
POST   /api/projects/:id/files                문서 파일 업로드→즉시 인덱싱
```

인증: Supabase Auth(이메일 초대) + RLS. 워커→DB는 service role 키.
