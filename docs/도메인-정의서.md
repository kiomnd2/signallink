## 1. 도메인 지도 (Bounded Contexts)

시그널링크는 **모듈러 모놀리스**다. 도메인 5개는 Gradle 모듈로 코드 경계를 갖되, 배포는 컨테이너 2개(`app-collector`·`app-api`)로만 한다.

```
                   ┌─────────────────────────────────────────────┐
                   │                domain-insight                │  ⬜
                   │   특징주 선정 · Why 3줄 엔진 · 체력 진단 · LLM   │
                   └───────────┬───────────┬───────────┬─────────┘
                     읽기(ID)  │           │           │  읽기(ID)
                 ┌─────────────┘           │           └─────────────┐
                 ▼                         ▼                         ▼
        ┌────────────────┐      ┌────────────────┐        ┌────────────────┐
        │  domain-macro  │ ⬜   │  domain-issue  │ 🟩🟨   │  domain-market │ 🟨
        │ 지표·신호등·이벤트 │      │  공시(DART)·뉴스   │        │  시세·수급(KIS)   │
        └───────┬────────┘      └────────────────┘        └────────────────┘
                │ 읽기(ID)                                          ▲
                └──────────────────────────────────────────────────┘

        ┌────────────────┐
        │ domain-content │ ⬜   용어사전 · 도미노 룰   (완전 독립)
        └────────────────┘
```

**의존 방향(단방향, 빌드에서 강제)**

| 도메인 | 의존(읽기 전용) | 피의존 | 비고 |
|--------|----------------|--------|------|
| `domain-market`  | 없음 | issue, macro, insight | KIS 앱키 소유 예정 |
| `domain-issue`   | 없음 | insight | KIS와 완전 독립 (DART·네이버만) |
| `domain-macro`   | market | insight | |
| `domain-insight` | market, issue, macro | 없음 | LLM 키 소유 예정 |
| `domain-content` | 없음 | 없음 | 완전 독립 |

**경계 규칙 (전 도메인 공통)**

1. **쓰기는 소유 도메인만** — 각 도메인은 자기 스키마(`market.*`, `issue.*`, …)에만 write.
2. **도메인 간 참조는 ID만** — 다른 도메인 테이블에 FK를 걸지 않는다. 필요한 식별자(`stock_code` 등)만 값으로 보관.
3. **역방향 참조는 컴파일 에러** — Gradle 모듈 의존으로 강제. 조합(여러 도메인 데이터 결합)은 상위(`domain-insight` 또는 `app-*`)에서.
4. **외부 기술은 포트 경유** — DB·HTTP·타 도메인 데이터는 `application/port/out` 인터페이스를 통해서만 접근하고, 구현은 `adapter/`에만 둔다 (헥사고날 Level 1, `ARCHITECTURE.md`).

## 2. 공통 언어 (Ubiquitous Language)

전 도메인이 공유하는 핵심 어휘. 코드·문서·대화에서 같은 뜻으로만 쓴다.

| 용어 | 정의 |
|------|------|
| **종목(stock)** | 상장 주식. 전 도메인의 연결 키는 6자리 **`stock_code`** (예: `005930`). FK가 아닌 값으로만 전파된다. |
| **공시(disclosure)** | 기업이 DART에 제출한 법정 공시. `rcept_no`(접수번호)가 자연 유니크 키. |
| **뉴스(news)** | 종목명으로 수집한 네이버 뉴스 기사. `(stock_code, url)`이 유니크. ⬜ |
| **특징주** | 등락률 ±5% 이상 또는 거래량 20일 평균 3배 이상 → 거래대금 상위 40종목. insight가 선정. ⬜ |
| **신호등** | 지표 발표를 🟢🟡🔴로 판정한 결과 (발표치 vs 예상치 vs 이전치 3단 비교). macro 소관. ⬜ |
| **매크로 이벤트(macro_event)** | 실제 지표 이벤트 이력. 신호등 판정 시 자동 기록. ⬜ |
| **자산 반응(asset_reaction)** | 이벤트 후 D+1/D+7/D+30 자산별 실측 등락률. ⬜ |
| **도미노 룰(domino_rule)** | 거시 이벤트의 인과 전파 규칙 `{from, to, direction, strength, ...}`. content 소관, DB로 관리. ⬜ |
| **Why 3줄 보드** | 급등락 원인을 ①무슨 일 ②수급 ③맥락 3줄로 번역한 카드. insight 소관. ⬜ |
| **워치리스트(watchlist)** | 뉴스/수집 대상 종목 목록. 초기엔 시드 CSV, 나중에 특징주로 교체 예정. ⬜ |
| **잠정 / 확정** | 장중 배치가 넣은 잠정치는 장 마감 확정 배치가 덮어쓴다. |
| **KST** | 모든 시각의 기준 타임존. `common.TimeUtils`가 담당 (`OffsetDateTime` ↔ `timestamptz`). |

## 3. 도메인별 정의 (개괄)

각 도메인의 *책임 / 소유 데이터 / 핵심 개념 / 외부 의존 / 현황*. 상세 스키마·엔티티는 실구현 시 각 도메인의 정의서/플랜에서 확장한다.

### 3.1 domain-market — 시세·수급 🟨

| 항목 | 내용 |
|------|------|
| 책임 | 종목 시세·등락률·투자자별(개인/외국인/기관) 순매수 등 **원천 시장 데이터 수집·보관** |
| 소유 스키마 | `market.*` (예정: 종목 마스터, 시세, 투자자 매매) |
| 외부 의존 | **KIS(한국투자증권) OpenAPI** — 앱키 소유 도메인 |
| 다른 도메인 참조 | 없음 (최하위 원천) |
| 현황 | `Stock`·`StockRepository`가 있으나 **예시/미구현 placeholder** (`ARCHITECTURE.md` §7). KIS 실키(M0) 이후 헥사고날 구조로 재배치 예정 |
| 분리 신호 | KIS 호출 한도로 별도 IP 필요 시 컨테이너 분리 검토 |

### 3.2 domain-issue — 공시·뉴스 🟩🟨

| 항목 | 내용 |
|------|------|
| 책임 | DART 신규 공시 폴링·분류, 네이버 뉴스 수집. 종목별 "이슈" 원천 데이터 |
| 소유 스키마 | `issue.disclosure` 🟩 · `issue.news` ⬜ |
| 외부 의존 | **OpenDART**(공시), **네이버 검색 API**(뉴스) — 둘 다 무료·즉시. **KIS와 완전 독립** |
| 다른 도메인 참조 | 없음. `stock_code`만 값으로 보관 (market이 비어도 적재 가능) |
| 현황 | **공시(Disclosure) 수집 슬라이스 구현 완료** 🟩 / 뉴스·워치리스트·수집 스케줄러 잡 ⬜ |
| 상세 | **→ §4** (이 프로젝트 헥사고날 Level 1의 첫 레퍼런스 구현) |

### 3.3 domain-macro — 지표·신호등·이벤트 ⬜

| 항목 | 내용 |
|------|------|
| 책임 | 경제 지표(미·한 14종) 신호등 판정, 매크로 이벤트 이력·자산 반응 배치 |
| 소유 스키마 | `macro.*` (예정: `macro_event`, `asset_reaction`, 지표·예상치) |
| 외부 의존 | FRED API(미 고용 등), 통계청 KOSIS API(한국 고용동향). 예상치는 관리자 수동 입력 |
| 다른 도메인 참조 | **market 읽기(ID)** — 자산 반응 산출에 시세 필요 |
| 현황 | `package-info`만. 미구현 |

### 3.4 domain-insight — 특징주·Why 엔진·진단 ⬜

| 항목 | 내용 |
|------|------|
| 책임 | 특징주 선정 → 시장 분해·수급 진단 → 공시·뉴스 매칭 → **Why 3줄 LLM 요약** + 장기 체력 진단(경량판) |
| 소유 스키마 | `insight.*` (예정: 특징주, Why 카드, 진단 결과) |
| 외부 의존 | **LLM(Gemini 3 Flash)** — 키 소유 도메인. 출력 필터(조언성 문구 차단) 필수 |
| 다른 도메인 참조 | **market·issue·macro 읽기(ID)** — 최상위 조합 도메인 |
| 현황 | `package-info`만. 미구현. 규제상 매수/매도/목표가 생성 금지(이중 차단) |
| 분리 신호 | LLM 배치가 조회 응답에 영향 줄 때 컨테이너 분리 검토 |

### 3.5 domain-content — 용어사전·도미노 룰 ⬜

| 항목 | 내용 |
|------|------|
| 책임 | 금융 번역기 용어사전(`glossary`, 300개), 매크로 도미노 인과 룰(`domino_rule`, ~30개) — **배포 없이 관리자 CRUD로 수정** |
| 소유 스키마 | `content.*` (예정: `glossary`, `domino_rule`) |
| 외부 의존 | 없음 |
| 다른 도메인 참조 | **없음 (완전 독립)** |
| 현황 | `package-info`만. 미구현 |

## 4. domain-issue 상세 정의

> `docs/ARCHITECTURE.md`의 헥사고날 Level 1(실용)을 적용한 **첫 레퍼런스 구현**. 아웃바운드 포트는 필수, 인바운드 포트(`port/in`)는 진입점이 수집 잡 하나뿐이라 생략(§4.6).

### 4.1 책임과 경계

- **한다**: DART에서 최근 공시를 폴링 → 상장사·신규만 선별 → 카테고리 분류 → `issue.disclosure` 저장. (뉴스 수집은 계획 ⬜)
- **안 한다**: 종목 시세 조회(market), 공시-특징주 매칭·요약(insight). 자기 스키마 밖 write 금지.
- **연결**: `stock_code`(6자리)만 값으로 보관. `market.stock`에 FK 없음 → market이 비어 있어도 독립 적재.

### 4.2 도메인 모델 (Entity)

#### `Disclosure` 🟩 — 공시 (Aggregate Root)

`issue.disclosure` 테이블 매핑. 헥사고날 Level 1 원칙에 따라 **JPA 엔티티 = 도메인 모델 겸용**(별도 매퍼 없음).

| 필드 | 타입 | 컬럼 | 제약 / 의미 |
|------|------|------|-------------|
| `id` | Long | `id` bigserial | PK (IDENTITY) |
| `stockCode` | String | `stock_code` varchar(6) | NOT NULL. 종목 연결 키(값 참조) |
| `receiptNo` | String | `rcept_no` varchar(20) | **UNIQUE**, NOT NULL. DART 접수번호 = **중복 수집 방지 자연 키** |
| `title` | String | `title` varchar(300) | NOT NULL. DART `report_nm` |
| `category` | String | `category` varchar(30) | nullable. 분류기 산출값(§4.3) |
| `disclosedAt` | OffsetDateTime | `disclosed_at` timestamptz | NOT NULL. KST 기준 |
| `dartUrl` | String | `dart_url` varchar(300) | NOT NULL. 공시 원문 뷰어 링크 |

- **불변식**: `receiptNo`는 전역 유일 → 같은 공시는 한 번만 저장(멱등성의 근거).
- **생성**: 공개 생성자(6개 필드) + JPA 전용 `protected` 기본 생성자. 세터 없음(수집 후 불변).

#### `News` ⬜ — 뉴스 (계획)

`issue.news` 테이블 예정: `(id, stock_code, title, summary, url, source, published_at)`, **`UNIQUE(stock_code, url)`** 로 중복 1차 방어. 미구현.

### 4.3 도메인 규칙 (Domain Rule)

#### `DisclosureCategoryClassifier` 🟩 — 공시 카테고리 분류

프레임워크 비의존 **순수 로직**(단독 단위 테스트 가능). 공시 제목(`report_nm`) 키워드로 카테고리를 매긴다. **위에서부터 먼저 걸리는 규칙 채택**, 미매칭은 `기타`.

| category | 키워드 |
|----------|--------|
| `수주계약` | 공급계약, 수주, 단일판매 |
| `실적` | 잠정실적, 영업(잠정), 매출액또는손익, 실적 |
| `자금조달` | 유상증자, 전환사채, 신주인수권, 교환사채 |
| `소송` | 소송, 분쟁 |
| `지분변동` | 최대주주, 대량보유, 주요주주, 지분 |
| `기타`(ETC) | (매칭 없음 / 제목 null) |

> 규칙은 스펙 확정 시 확장 대상. 배포 없이 바꾸려면 향후 DB화 검토(현재는 코드 상수).

### 4.4 경계 타입 (게이트웨이 DTO)

#### `DisclosureCandidate` 🟩 — 가공 전 공시 후보 (record)

외부 API 응답 형식과 도메인을 분리하는 경계 타입. 게이트웨이가 돌려주고, 카테고리 분류·중복 판별·엔티티 변환은 **유스케이스**가 수행.

```
DisclosureCandidate(receiptNo, stockCode, title, disclosedAt, dartUrl)
```
- `stockCode`: 상장사만 값 존재, 비상장은 blank → 유스케이스가 필터링.

### 4.5 포트 (Ports)

#### 아웃바운드 포트 (`application/port/out`) — 필수

| 포트 | 계약 | 격리 대상 | 구현 어댑터 |
|------|------|-----------|-------------|
| `DartGatewayPort` 🟩 | `List<DisclosureCandidate> fetchDisclosures(from, to)` | 외부 API(DART) | `DartClient` |
| `DisclosureRepositoryPort` 🟩 | `boolean existsByReceiptNo(no)` · `Disclosure save(d)` | DB(JPA) | `DisclosurePersistenceAdapter` |
| `NaverNewsGatewayPort` ⬜ | (뉴스 조회) | 외부 API(네이버) | `NaverNewsClient` |
| `NewsRepositoryPort` ⬜ | (뉴스 영속) | DB(JPA) | `NewsPersistenceAdapter` |
| `WatchlistPort` ⬜ | (뉴스 대상 종목) | 종목 소스 | `SeedWatchlistAdapter`(시드 CSV) → 후에 특징주로 교체 |

> 포트의 이득: 서비스 단위 테스트 시 외부 호출/DB 없이 **스텁으로 대체**. `WatchlistPort`는 시드 → 특징주(market) 소스를 **어댑터 교체만으로** 전환하는 지점.

#### 인바운드 포트 (`application/port/in`) — 생략

수집 잡이 유일 진입점이라 `port/in` 없이 `@Service`를 인바운드 어댑터가 직접 호출(`ARCHITECTURE.md` §5 실용 규칙). 여러 진입점이 생기면 `UseCase` 인터페이스로 승격.

### 4.6 유스케이스 (Application Service)

#### `DisclosureCollectService` 🟩 — 공시 수집

아웃바운드 포트에만 의존(생성자 주입). 흐름:

```
collectRecent(daysBack):
  1. DART에서 최근 daysBack일 공시 조회         (DartGatewayPort)
  2. stock_code 없으면 스킵                     (비상장/시장 외)
  3. rcept_no 이미 있으면 스킵                   (DisclosureRepositoryPort — 멱등)
  4. 제목으로 category 분류                       (DisclosureCategoryClassifier)
  5. Disclosure 저장 → 신규 저장 건수 반환
```

- **멱등성**: 재실행해도 `rcept_no` 기준 중복 저장 없음 → 폴링 주기를 겹치게 돌려도 안전.
- `NewsCollectService` ⬜ — 워치리스트 주입받아 종목별 뉴스 수집(계획).

### 4.7 어댑터 (Adapters)

| 어댑터 | 위치 | 역할 |
|--------|------|------|

### 4.8 데이터 모델 (issue 스키마)

```sql
-- 🟩 구현됨 (운영 스키마 V1__init.sql의 issue.disclosure와 동일 유지)
CREATE TABLE issue.disclosure (
  id           bigserial PRIMARY KEY,
  stock_code   varchar(6)  NOT NULL,
  rcept_no     varchar(20) UNIQUE NOT NULL,   -- 중복 수집 방지 자연 키
  title        varchar(300) NOT NULL,
  category     varchar(30),
  disclosed_at timestamptz NOT NULL,
  dart_url     varchar(300) NOT NULL
);
-- idx: (stock_code, disclosed_at)

-- ⬜ 계획
issue.news (id, stock_code, title, summary, url, source, published_at)
  UNIQUE (stock_code, url)   -- 중복 제거 1차 방어
  idx: (stock_code, published_at)
```

- `ddl-auto: none` — 스키마는 마이그레이션(Flyway, `app-api` 소유)이 관리. 엔티티는 매핑만.
