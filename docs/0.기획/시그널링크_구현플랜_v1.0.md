# 시그널링크 구현 플랜 v1.0

기준: 기획확정서 v2.5 + 백엔드 상세스펙 v1.0 | 1인 개발, 13~15주
원칙: **마일스톤마다 "돌아가는 것"을 만든다** — 각 단계 종료 시 데모 가능한 상태 유지. 이전 마일스톤의 Exit 조건을 통과해야 다음으로 진행.

## M0. 준비·검증 (1주차) — 최대 리스크 제거

**계정·키 (Day 1~2)**
- [ ] 한국투자증권 계좌 개설 → KIS Developers 모의·실전 앱키
- [ ] OpenDART, 네이버 검색, FRED, ECOS, KOSIS, Gemini 키 발급
- [ ] LLM 품질 비교(반나절): 동일 특징주 3건을 Gemini 3 Flash(기본)·Flash-Lite·Haiku로 요약 → 한국어 품질·금지어 준수 확인
- [ ] GitHub repo(private), Supabase 프로젝트, 국내 VPS(4GB) 계약

**KIS 실검증 (Day 2~5) — 임시 스크립트로, 프로젝트 구조와 무관하게 빠르게**
- [ ] 토큰 발급 → 등락률·거래량 순위 TR → 종목별 투자자 동향 TR 실호출
- [ ] TR ID·응답 필드를 상세스펙 5장 표에 기입 (스펙 확정)
- [ ] 신규 고객 호출 한도 실측 → J2(30분 배치, 40종목) 소요 호출 수 계산 → 한도 내 확인
- [ ] **VPS에서 동일 호출 성공 확인** (IP 차단 여부 — 실패 시 국내 타 VPS로 즉시 교체)
- [ ] LLM: 실제 특징주 1건을 스펙 6장 프롬프트로 요약 → 출력 필터 통과 확인

**Exit**: 위 체크 전부 통과. ⚠️ KIS 한도가 부족하면 여기서 배치 주기(30분→60분)나 종목 수(40→25)를 조정하고 확정서에 반영 — 뒤에서 발견하면 재작업 비용 10배.

## M1. 골격 (2주차) — 이후 전부가 올라갈 기반

- [ ] Gradle 멀티모듈 생성: common / domain-{market,issue,macro,insight,content} / app-{collector,api}
- [ ] 의존 방향 강제: Gradle 모듈 의존 선언만으로 역방향 참조가 컴파일 에러가 되는지 확인
- [ ] Flyway 도입 → 상세스펙 2장 DDL을 V1__init.sql로 (스키마 5개)
- [ ] Dockerfile 2개 + docker-compose.yml (로컬: 앱 2 + 로컬 postgres)
- [ ] GitHub Actions: PR 시 빌드+테스트 / main 머지 시 이미지 빌드 → VPS 배포 (ssh + compose pull&up)
- [ ] 공통 모듈: KST 시간 유틸, 공통 응답 포맷, 외부 API 클라이언트 베이스(rate limiter, 백오프 재시도), Discord 알림
- [ ] /actuator/health → VPS 배포 → Uptime Kuma 연결

**Exit**: 빈 API가 VPS에서 응답하고, git push만으로 배포되는 상태.
**팁**: 여기서 완벽주의 금지 — 관측·백업 등은 M7에서 보강. 2주차는 "배포 파이프라인이 도는 것"이 전부.

## M2. 원천 데이터 (3~4주차) — domain-market, domain-issue

구현 순서 (의존 순):
1. [ ] market: 종목 마스터 동기화(J1) → daily_price 수집 + **2년치 백필** (일봉, 지수 포함 — 도미노 리플레이·베타 계산의 원천)
2. [ ] market: investor_flow 수집 (장중 잠정 + 마감 확정), vol_ratio_20d 계산
3. [ ] market: stock_beta 주간 계산(J7) — 백필 데이터로 즉시 검증 가능
4. [ ] issue: DART 신규 공시 폴링 + 카테고리 분류(제목 키워드 룰)
5. [ ] issue: 네이버 뉴스 수집 (종목명 쿼리, 중복 URL 제거)
6. [ ] collector 스케줄러에 J1/J3/J7 등록, 장 운영일 캘린더(휴장일) 처리

**Exit**: 평일 하루 무인 방치 후 daily_price·investor_flow·disclosure·news에 당일 데이터가 정상 적재. 잠정→확정 덮어쓰기 확인.
**테스트 전략**: 외부 API 클라이언트는 실응답 JSON을 픽스처로 저장해 파서 단위 테스트. 수집 로직은 통합 테스트 대신 "하루 돌려보고 데이터 검증" 스크립트(row 수·null 비율 체크)로 실용적으로.

## M3. Why 엔진 (5~7주차) — domain-insight, 서비스의 심장

1. [ ] 특징주 선정 쿼리 (스펙 3.1) — 백필 데이터로 과거 날짜 재현 테스트 ("그날 특징주가 맞게 뽑히나")
2. [ ] 시장분해·업종판별 (3.2의 1·2단계) → 템플릿 문장 생성기
3. [ ] 수급 진단 (4단계) → flow_summary 생성
4. [ ] 이슈 매칭 (3단계): 공시 우선, 뉴스 시간창 필터
5. [ ] LLM 클라이언트: 페이로드 조립 → 호출 → JSON 파싱 → 금지어 필터 → 실패 시 템플릿 폴백 → 일 상한 가드
6. [ ] J2 파이프라인 조립: 선정→분석→카드 upsert, 종목 단위 예외 격리
7. [ ] health_check: 이슈 성격 태그(공시 카테고리 룰 + LLM 보조) + DART 재무 추세 + flow_after 추적(J6)
8. [ ] macro: macro_event 기록 + asset_reaction 계산(J5) + **2년치 이벤트 백필 스크립트**

**Exit**: 장중 30분마다 카드 40개가 자동 생성·갱신되고, LLM 출력 20건 수동 검수에서 조언성 문구 0건. 과거 유명 급등락 사례 3건(예: 대형 수주 공시일)을 백필로 재현해 "사람이 봐도 맞는 설명"인지 확인.
**주의**: 이 마일스톤이 가장 밀리기 쉬움 — 7주차 말 기준 5번(LLM)까지만 되어 있어도 M4로 진행하고 7·8번은 병행.
