## 1. 확정 범위

### 1차 출시 기능 (4개)

| # | 기능 | 한 줄 정의 |
|---|------|------|
| F1 | 매크로 도미노 시뮬레이터 + 과거 리플레이 | 거시 이벤트의 파급 경로를 이론(룰)과 실측(과거 데이터)으로 시각화 |
| F2 | 금융 번역기 | 어려운 용어를 터치 한 번에 일상어로 번역 (DB 관리, 인기 용어 집계) |
| F3 | 경제 신호등 + 시장 날씨 | 지표 발표를 🟢🟡🔴 신호와 날씨 배경으로 직관화 |

### 확정 제외

투자 성향 진단 — 기능 자체 제외 확정.
종목 추천 — 형태 불문 미도입. "조건 공개형 발견 도구"로만 접근(2차 F6).

### 2차 로드맵 (우선순위 순)

1. **F6. 정세→테마→종목 연결 맵 (최우선, 출시 후 2~3주)** — 국내·글로벌 정세 이벤트 발생 시 도미노 룰로 수혜/피해 업종·테마를 표시하고, 관련 종목을 실측 데이터 순위(외국인·기관 순매수, 거래대금 급증 — 조건 화면 공개)로 나열. "추천"이 아닌 사실 서술형. 1차 산출물(domino_rule, macro_event, asset_reaction, investor_flow) 전부 재활용. 기존 기획의 시그널 레이더는 이 기능에 흡수.
2. **F7. 실적 시즌 번역기** — 어닝 서프라이즈/쇼크를 컨센서스 대비로 번역하고 "실적이 좋은데 왜 떨어져?"(선반영) 개념을 사례로 학습. DART 실적 공시 + Why 보드 파이프라인 재활용. 분기마다 반복되는 리텐션 이벤트.
3. **F8. 투자 시뮬레이터 (과거 + 미래 시나리오)**
   - 과거: "3년 전부터 매달 10만 원 적립했다면 vs 몰빵했다면 vs 예금했다면" — 실측 데이터 계산, 적립·분산·장기의 효과를 숫자로 체감
   - 미래: 예측이 아닌 **가정 명시형 시나리오 계산기** — 사용자가 시나리오 선택(과거 10년 평균 반복 / 최악의 해 반복 / 예금 금리), 복리·적립 효과 산출. "과거 수익률은 미래를 보장하지 않습니다" 상시 고지, 특정 종목 기대수익 단정 표현 금지
   - 결과 화면은 공유 카드와 연계 (바이럴 성장 채널)
4. 관심종목 데일리 브리핑 (회원 기능과 함께)
5. 과거 타임머신 (수동 사례 DB 방식) — 완성 시 장기 체력 진단 카드의 '유사 사례 회복/미회복 통계'가 활성화됨
6. 학습 스트릭·퀴즈, 공유 카드, 뇌동매매 자기진단(행동재무 시나리오 퀴즈)
7. 알림톡·캘린더 배달 (사업자등록 후)
8. 스크리닝(장기 발견 도구), 스트레스 테스트

## 2. 기능 상세 명세 (1차)

### F1. 매크로 도미노 시뮬레이터 + 과거 리플레이

- 트리거 이벤트 10개: 금리 인상/인하, 물가 상승/하락, 환율 상승/하락, 유가 상승/하락, **고용 호조/부진** (고용→연준 금리 경로의 2단 도미노 — "좋은 뉴스가 나쁜 뉴스" 케이스 학습에 최적)
- 자산 노드 6개: 채권, 달러, 국내 기술주, 수출주, 금, 부동산
- 인과 룰 약 30개를 DB(`domino_rule`)로 관리: `{from, to, direction, strength, explanation, exception_case}` — 배포 없이 수정 가능, 룰마다 예외 사례 카드 1개 이상 필수
- **과거 리플레이**: 시뮬레이션(이론) 옆 "실제로는 어땠나" 탭
  - `macro_event`: 실제 지표 이벤트 이력 — collector가 신호등 판정 시 자동 기록
  - `asset_reaction`: 이벤트 후 D+1/D+7/D+30 자산별 실측 등락률 — 후속 배치가 자동 채움
  - 오픈 시 최근 2년치 이벤트 백필
  - 예: "이론상 금리 동결은 기술주 상승 신호지만, 지난 3번의 실제 결과는 +2.1% / −0.8% / +4.3%였습니다"
- 구현: SVG + CSS 애니메이션, 클릭 시 연쇄 전파

### F2. 금융 번역기

- 런칭 시 용어 300개 (거시/실적/수급/차트/공시)
- DB(`glossary`) + 관리자 CRUD — 배포 없이 추가·수정
- 스키마: `{term, easy_explanation(2문장), example_sentence, related_terms, category, view_count}`
- `view_count`로 "이번 주 많이 찾은 용어" 위젯 + 보강 우선순위 판단
- 화면 내 용어 자동 감지 → 점선 밑줄 → 터치 시 말풍선 (프론트 캐시, 1일 1회 갱신)
- '초보자 필터' 토글 (기본 on)

### F3. 경제 신호등 + 시장 날씨

- 지표 14개
  - 미국: CPI·PCE·FOMC·GDP + **고용 3종 세분화** — 비농업고용(NFP)·실업률·시간당 임금 (발표 동시, 시장 영향 최대급 → 신호등 개별 판정 후 종합 신호 1개로 표시)
  - 한국: CPI·금통위·수출입 + **고용동향** (통계청 — 취업자 수 증감·실업률, 매월 중순 발표)
  - 시장: 미 10년물, 달러인덱스, VIX, 원/달러
- 초보자 해설 예시: "미국 일자리가 예상보다 많이 늘면(고용 호조) 금리 인하가 늦어질 수 있어 주식엔 🟡 — '좋은 뉴스가 나쁜 뉴스'가 되는 대표 사례입니다" — 고용지표는 해석이 반직관적이라 번역기·도미노와 연계 효과가 큼
- 데이터 소스: 미 고용 FRED API(무료), 한국 고용동향 통계청 KOSIS API(무료)
- 판정 룰: 발표치 vs 예상치 vs 이전치 3단 비교 → 🟢🟡🔴 + 한 줄 해설 템플릿
- 예상치는 발표 전 관리자 폼에서 수동 입력 (크롤링 약관 리스크 회피)
- 시장 날씨: KOSPI 등락률 + VIX 조합 5단계(화창~폭풍) → 대시보드 배경 테마

### F5. 특징주 Why 3줄 보드 — 핵심 기능

**특징주 선정 기준**: 등락률 ±5% 이상 또는 거래량 20일 평균 대비 3배 이상 → 거래대금 상위 40종목. 장중 30분 배치 + 장 마감 확정 배치.

**Why 엔진 — 4단계 소거 로직 (룰 기반, LLM 비용 0)**

1. 시장 분해: 종목 등락률 − (베타 × 지수 등락률)
2. 업종 판별: 동일 업종지수 동반 등락 여부
3. 개별 이슈 매칭: 등락 시각 전후 DART 공시 + 네이버 뉴스 후보 3건
4. 수급 진단: 개인/외국인/기관 순매수, 거래량 배율, 체결강도 (장중 "잠정" 표기)

**LLM 요약** (뉴스가 있는 종목만, 일 20~30건)

- 입력: 룰 엔진 산출물 + 기사 3건 → AI는 검증된 데이터의 "통역"만 담당 (환각 방지)
- 출력 고정 포맷: ①무슨 일 ②수급(수치 병기) ③맥락(시장 분해)
- 차단 규칙: 매수/매도/보유 권유·목표가·미래 전망 생성 금지 — 프롬프트 + 출력 필터 이중 차단
- 뉴스 없는 종목은 템플릿 문장 처리

**카드 예시**

```
📉 한화오션 −8.2%  [잠정]
① 무슨 일: 오후 2시 잠수함 수주 불발 공시 → 원문 링크
② 수급: 기관 −30만 주 주도 · 거래량 평소 4배 · 개인 +25만 주
③ 맥락: 이 중 −2%는 시장 하락분
🔍 장기 체력 진단 펼쳐보기 ▼ | 본 정보는 투자 조언이 아닙니다
```

**장기 체력 진단 카드 (경량판) — "이 악재, 오래 갈 이슈인가?"**

"보유해라/팔아라" 직접 조언은 규제·신뢰 양면에서 미도입 확정. 대신 장기 관점 판단에 필요한 팩트를 제공하고 판단은 사용자가 한다.

1. **이슈 성격 태그**: 일회성 사건(수주 불발·소송·단기 수급)인지 구조적 문제(적자 전환·주력사업 규제)인지 분류 — 룰 기반 분류 + LLM 보조, 애매하면 "분류 불가" 표시
2. **실적 추세**: DART 재무 데이터로 최근 4분기 매출·영업이익 방향 (개선/유지/악화)
3. **수급 추세**: 이슈 발생 후 기관·외국인이 계속 매도 중인지, 매도가 멈췄는지 (D+1부터 자동 추적)
4. 과거 유사 사례 회복/미회복 통계 — **2차** (타임머신 사례 DB 필요, 회복·실패 사례 반드시 병기)

출력 원칙: "장기 투자자가 확인할 팩트"까지만 제시, 결론 문장(보유·매도 권유) 생성 금지 — F5 차단 규칙 동일 적용.

## 3. 시스템 아키텍처

**방침**: **모듈러 모놀리스** — 도메인 5개는 Gradle 모듈로 분리하되, 배포는 컨테이너 2개(collector/api)만. 도메인 경계는 코드에 존재하므로 트래픽이 생기면 그때 개별 도메인을 컨테이너·인스턴스로 분리할 수 있다. 1인 운영에서 JVM 5개(풀 컨테이너 분리)는 운영 부담이 이득보다 크다고 판단, 미도입 확정.

```
[Gradle 도메인 모듈 — 코드 경계]
 domain-market   시세·등락률·투자자별 수급          [KIS 키]
 domain-issue    DART 공시·네이버 뉴스·종목-이슈 매칭
 domain-macro    지표·신호등·macro_event·자산반응
 domain-insight  특징주 선정·Why 엔진·체력 진단·LLM  [LLM 키]
 domain-content  용어사전·도미노 룰

[배포 — VPS 1대 (2vCPU/4GB), Docker Compose 컨테이너 2개]
 ├─ app-collector  스케줄러: market·issue·macro·insight 조립
 │    └─ 장중 30분 / 장 마감 확정 / 신호등 판정 / D+1/7/30 자산반응 배치
 └─ app-api        REST: 전 도메인 조회 + /admin/** (관리자 인증)
      └─ /features /health-check /signals /glossary /domino/rules /domino/replay

PostgreSQL (Supabase) — 도메인별 스키마: market.* / issue.* / macro.* / insight.* / content.*
Next.js 프론트 (Vercel) — app-api만 호출
 ├─ / (대시보드: 날씨 배경 + 신호등 + 특징주 보드)
 ├─ /domino (시뮬레이터 + 과거 리플레이 탭)
 └─ 공통: 금융 번역기, 초보자 필터
```

**도메인 의존 규칙 (단방향)**: insight → market·issue·macro (읽기 전용) / macro → market / content → 독립. 쓰기는 소유 도메인만, 도메인 간 참조는 ID만.

- **스택**: Java 21 + Spring Boot 3.x / Gradle 멀티모듈(도메인 5 + 공통 + 앱 2) / Docker Compose / PostgreSQL / Next.js + Tailwind / LLM: Gemini 3 Flash
- **배포**: VPS 1대(2vCPU/4GB), CI는 GitHub Actions
- **원칙**: ①실시간 아님(30분 배치) ②KIS·LLM 키는 collector에만 ③확정 배치가 잠정치 덮어씀 ④도메인 모듈 경계 위반은 빌드에서 차단(의존 방향 강제)
- **추후 분리 신호(그 전엔 분리하지 않음)**: insight — LLM 배치가 조회 응답에 영향 줄 때 / market — KIS 호출 한도로 별도 IP 필요 시 / 조회 트래픽 증가는 캐시·수평 복제로 대응
- **2차 확장**: domain-user·domain-notify·domain-thememap(F6) 모듈 추가 — 필요해질 때만 컨테이너 분리·게이트웨이·Kafka 도입

## 5. 개발 일정 (13~15주)

| 주차 | 작업 | 산출물 |
|------|------|------|
| 1 | Gradle 도메인 멀티모듈 셋업(domain 5 + app 2), API 키 발급, KIS API 검증(토큰→등락률→투자자별 매매) | 수집 PoC + 골격 |
